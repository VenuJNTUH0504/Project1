Error Handling (try/catch)

Error handling is a critical aspect of writing robust JavaScript applications. It allows you to gracefully handle unexpected situations, provide meaningful feedback to users, and prevent your application from crashing when something goes wrong.

Types of Errors in JavaScript

JavaScript has several built-in error types:

  • SyntaxError: Occurs when there's a mistake in the syntax of your code
  • ReferenceError: Occurs when referencing a variable that doesn't exist
  • TypeError: Occurs when a value is not of the expected type
  • RangeError: Occurs when a value is not in the expected range
  • URIError: Occurs when using global URI handling functions incorrectly
  • EvalError: Occurs when using the eval() function incorrectly
  • Error: The generic error object, base for all other error types

Examples of Common Errors

// SyntaxError
// const user = { name: "John", age: 30; }  // Missing comma, using semicolon

// ReferenceError
console.log(undefinedVariable);  // Variable doesn't exist

// TypeError
const str = "Hello";
str.push("World");  // String doesn't have a push method

// RangeError
const arr = new Array(-1);  // Cannot create array with negative length

The try/catch Statement

The try/catch statement allows you to "try" running some code and "catch" potential errors without crashing your program.

Basic Syntax

try {
  // Code that might throw an error
} catch (error) {
  // Code to handle the error
}

Simple Example

try {
  // Attempt to access a property of an undefined variable
  console.log(nonExistentVariable.property);
} catch (error) {
  console.error('An error occurred:', error.message);
  // Output: "An error occurred: Cannot read properties of undefined (reading 'property')"
}

The Error Object

When an error occurs, JavaScript creates an Error object with properties that provide information about the error:

  • name: The name of the error (e.g., "ReferenceError")
  • message: A human-readable description of the error
  • stack: A stack trace showing where the error occurred
try {
  throw new Error('Something went wrong');
} catch (error) {
  console.log('Error name:', error.name);       // "Error"
  console.log('Error message:', error.message); // "Something went wrong"
  console.log('Error stack:', error.stack);     // Stack trace information
}

The finally Clause

The finally clause executes code after the try and catch blocks, regardless of whether an error occurred or not.

function processFile() {
  let file = null;
  
  try {
    file = openFile('example.txt');
    // Process the file
    return 'File processed successfully';
  } catch (error) {
    console.error('Error processing file:', error.message);
    return 'Error processing file';
  } finally {
    // This will run regardless of whether there was an error
    if (file) {
      closeFile(file); // Always close the file
      console.log('File closed');
    }
  }
}

When to Use finally

The finally block is particularly useful for cleanup operations that should happen regardless of success or failure, such as:

  • Closing file handles
  • Closing database connections
  • Releasing resources
  • Resetting state variables

Throwing Errors

You can manually throw errors using the throw statement. This is useful when you detect a problem in your code and want to signal that something went wrong.

function divide(a, b) {
  if (b === 0) {
    throw new Error('Division by zero is not allowed');
  }
  return a / b;
}

try {
  const result = divide(10, 0);
  console.log('Result:', result);
} catch (error) {
  console.error('Error:', error.message); // "Error: Division by zero is not allowed"
}

Creating Custom Error Types

You can create custom error types by extending the built-in Error class. This allows you to create more specific errors for your application.

// Define a custom error type
class ValidationError extends Error {
  constructor(message) {
    super(message);
    this.name = 'ValidationError';
  }
}

// Define another custom error type
class DatabaseError extends Error {
  constructor(message, code) {
    super(message);
    this.name = 'DatabaseError';
    this.code = code;
  }
}

// Using custom errors
function validateUser(user) {
  if (!user) {
    throw new ValidationError('User object is required');
  }
  
  if (!user.name) {
    throw new ValidationError('User name is required');
  }
  
  if (user.age < 18) {
    throw new ValidationError('User must be at least 18 years old');
  }
}

try {
  validateUser({ name: 'John', age: 16 });
} catch (error) {
  if (error instanceof ValidationError) {
    console.error('Validation failed:', error.message);
  } else if (error instanceof DatabaseError) {
    console.error(`Database error (${error.code}):`, error.message);
  } else {
    console.error('Unknown error:', error);
  }
}

Error Handling in Asynchronous Code

Promises

Promises have built-in error handling using the .catch() method.

function fetchData() {
  return new Promise((resolve, reject) => {
    // Simulate an API call
    setTimeout(() => {
      const success = Math.random() > 0.5;
      if (success) {
        resolve({ id: 1, name: 'John' });
      } else {
        reject(new Error('Failed to fetch data'));
      }
    }, 1000);
  });
}

fetchData()
  .then(data => {
    console.log('Data received:', data);
  })
  .catch(error => {
    console.error('Error:', error.message);
  })
  .finally(() => {
    console.log('Operation completed');
  });

Async/Await

With async/await, you can use traditional try/catch blocks for asynchronous code.

async function getUserData(userId) {
  try {
    const response = await fetch(`https://api.example.com/users/${userId}`);
    
    if (!response.ok) {
      throw new Error(`HTTP error! Status: ${response.status}`);
    }
    
    const data = await response.json();
    return data;
  } catch (error) {
    console.error('Error fetching user data:', error.message);
    // You can re-throw the error or return a default value
    return { error: true, message: error.message };
  }
}

// Using the function
async function displayUserProfile(userId) {
  const userData = await getUserData(userId);
  
  if (userData.error) {
    displayErrorMessage(userData.message);
  } else {
    displayUserInfo(userData);
  }
}

Common Mistake: Forgetting try/catch with await

A common mistake is forgetting to wrap await calls in try/catch blocks. Without proper error handling, unhandled promise rejections can crash your application.

Incorrect

async function fetchData() {
  // No error handling!
  const response = await fetch('/api/data');
  const data = await response.json();
  return data;
}

Correct

async function fetchData() {
  try {
    const response = await fetch('/api/data');
    const data = await response.json();
    return data;
  } catch (error) {
    console.error('Error:', error);
    throw error; // Re-throw or handle appropriately
  }
}

Best Practices for Error Handling

1. Be Specific About What You Catch

Catch specific errors rather than catching all errors indiscriminately.

try {
  // Code that might throw different types of errors
} catch (error) {
  if (error instanceof TypeError) {
    // Handle type errors
  } else if (error instanceof RangeError) {
    // Handle range errors
  } else if (error.name === 'ValidationError') {
    // Handle validation errors
  } else {
    // Handle other errors or re-throw
    throw error;
  }
}

2. Provide Meaningful Error Messages

Error messages should be clear and helpful, providing context about what went wrong.

Poor Error Message

if (!isValid) {
  throw new Error('Invalid input');
}

Better Error Message

if (!isValid) {
  throw new Error(
    'Invalid email format: ' + 
    email + 
    '. Email must contain @ symbol.'
  );
}

3. Log Errors Appropriately

Log errors with enough context to help with debugging, but be careful not to log sensitive information.

try {
  processUserData(userData);
} catch (error) {
  // Log with context
  console.error(
    `Error processing user data for user ${userData.id}:`, 
    error.message,
    { timestamp: new Date(), errorCode: error.code }
  );
  
  // Show user-friendly message
  displayErrorToUser('We encountered a problem processing your data.');
}

4. Don't Swallow Errors

"Swallowing" errors means catching them but not handling them properly, which can make debugging difficult.

Swallowing Errors (Bad)

try {
  riskyOperation();
} catch (error) {
  // Error is caught but ignored!
}

Proper Handling (Good)

try {
  riskyOperation();
} catch (error) {
  // Log the error
  console.error('Error in riskyOperation:', error);
  
  // Take appropriate action
  notifyUser();
  
  // Optionally re-throw if can't handle here
  throw error;
}

5. Clean Up Resources

Always clean up resources, even when errors occur, using finally blocks.

let connection = null;
try {
  connection = openDatabaseConnection();
  // Use the connection
} catch (error) {
  console.error('Database error:', error);
  // Handle the error
} finally {
  if (connection) {
    connection.close(); // Always close the connection
  }
}

Practical Examples

Form Validation with Error Handling

function validateForm(formData) {
  const errors = [];
  
  try {
    // Check required fields
    if (!formData.name) {
      errors.push('Name is required');
    }
    
    if (!formData.email) {
      errors.push('Email is required');
    } else if (!isValidEmail(formData.email)) {
      errors.push('Email format is invalid');
    }
    
    if (!formData.password) {
      errors.push('Password is required');
    } else if (formData.password.length < 8) {
      errors.push('Password must be at least 8 characters');
    }
    
    // If we have errors, throw a validation error
    if (errors.length > 0) {
      const error = new ValidationError('Form validation failed');
      error.errors = errors;
      throw error;
    }
    
    return { valid: true };
  } catch (error) {
    if (error instanceof ValidationError) {
      return { 
        valid: false, 
        errors: error.errors 
      };
    }
    
    // For unexpected errors
    console.error('Unexpected validation error:', error);
    return { 
      valid: false, 
      errors: ['An unexpected error occurred'] 
    };
  }
}

// Usage
const formData = {
  name: 'John',
  email: 'invalid-email',
  password: 'pass'
};

const result = validateForm(formData);

if (!result.valid) {
  // Display errors to the user
  result.errors.forEach(error => {
    console.log('- ' + error);
  });
} else {
  // Submit the form
  submitForm(formData);
}

API Error Handling

async function fetchUserData(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`);
    
    // Handle HTTP errors
    if (!response.ok) {
      const errorData = await response.json().catch(() => ({}));
      
      // Create appropriate error based on status code
      if (response.status === 404) {
        throw new NotFoundError(`User with ID ${userId} not found`);
      } else if (response.status === 401 || response.status === 403) {
        throw new AuthorizationError('Not authorized to access this user');
      } else {
        throw new ApiError(
          `API error: ${errorData.message || response.statusText}`,
          response.status
        );
      }
    }
    
    // Parse JSON (this could also throw)
    const data = await response.json();
    return data;
  } catch (error) {
    // Re-throw our custom errors
    if (error instanceof NotFoundError || 
        error instanceof AuthorizationError || 
        error instanceof ApiError) {
      throw error;
    }
    
    // Handle network errors
    if (error.name === 'TypeError' && error.message.includes('fetch')) {
      throw new NetworkError('Network error: Unable to connect to the server');
    }
    
    // Handle JSON parsing errors
    if (error instanceof SyntaxError && error.message.includes('JSON')) {
      throw new ApiError('Invalid response format from the server');
    }
    
    // Handle any other unexpected errors
    console.error('Unexpected error in fetchUserData:', error);
    throw new Error('An unexpected error occurred while fetching user data');
  }
}

// Usage with error handling
async function displayUserProfile(userId) {
  try {
    const userData = await fetchUserData(userId);
    renderUserProfile(userData);
  } catch (error) {
    if (error instanceof NotFoundError) {
      showNotFoundMessage();
    } else if (error instanceof AuthorizationError) {
      redirectToLogin();
    } else if (error instanceof NetworkError) {
      showOfflineMessage();
    } else {
      showGenericErrorMessage(error.message);
    }
  }
}

Error Boundary Pattern (Inspired by React)

class ErrorBoundary {
  constructor(fallbackFn) {
    this.fallbackFn = fallbackFn || (() => console.log('An error occurred'));
  }
  
  wrap(fn) {
    return async (...args) => {
      try {
        return await fn(...args);
      } catch (error) {
        console.error('Error caught by boundary:', error);
        return this.fallbackFn(error);
      }
    };
  }
}

// Usage
const errorBoundary = new ErrorBoundary((error) => {
  // Display a user-friendly error message
  document.getElementById('error-container').innerHTML = `
    

      
Something went wrong

      
${error.message}

      
    

  `;
  
  // Return a default value
  return { error: true, message: error.message };
});

// Wrap a function with error handling
const safeLoadData = errorBoundary.wrap(async () => {
  const response = await fetch('/api/data');
  if (!response.ok) throw new Error('Failed to load data');
  return response.json();
});

// Now you can use safeLoadData without try/catch
safeLoadData().then(data => {
  if (!data.error) {
    renderData(data);
  }
});
